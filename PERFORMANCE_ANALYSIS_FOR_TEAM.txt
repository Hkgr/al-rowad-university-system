SUMMARY (READ THIS FIRST)
-------------------------
The dashboard feels slow because of a COMBINATION of issues — not one single bug.

  - NOT mainly a database problem (only ~2 students in DB during testing)
  - YES backend server setup (php artisan serve = single-threaded)
  - YES frontend firing duplicate + too many API calls at once
  - YES React StrictMode doubles requests in development

Your friend may feel faster because he uses XAMPP Apache (concurrent PHP),
fewer API calls, mock/smaller data, or different pages — not because React
is inherently faster than Laravel.

WHAT THE HAR FILE SHOWED
================================================================================

Issue                          | Evidence
-------------------------------|-----------------------------------------------
Same API called many times     | GET /students?page=1&per_page=15 → 6 times
Student profile over-fetches   | 6 endpoints × duplicate runs = many GETs
Server wait time grows         | First request ~660ms → later ~2.6s (queued)
CORS preflight overhead        | Many OPTIONS requests waiting 400ms–5s each

Endpoints hit on student profile page (all at once on load):
  - GET /api/v1/students/{id}/profile
  - GET /api/v1/students/{id}/transcript
  - GET /api/v1/students/{id}/cgpa
  - GET /api/v1/students/{id}/attendance
  - GET /api/v1/academic-years
  - GET /api/v1/semesters

Average server wait (TTFB) when multiple calls stack:
  - /students list        → ~1.2s avg (6 calls)
  - /profile              → ~4.0s avg (4 calls)
  - /transcript           → ~4.2s avg
  - /cgpa                 → ~4.4s avg
  - /attendance           → ~4.6s avg
  - /academic-years       → ~4.7s avg
  - /semesters            → ~4.8s avg

On php artisan serve, 6 "parallel" Promise.all requests run ONE AT A TIME.
So total time ≈ sum of all requests, not the slowest one.


================================================================================
ROOT CAUSES (ORDERED BY IMPACT)
================================================================================

1. php artisan serve IS SINGLE-THREADED (BIGGEST BACKEND FACTOR)
   ----------------------------------------------------------------
   Frontend fires 4–6 parallel requests (Promise.all).
   Built-in PHP server handles ONE request at a time → they queue.

   Friend on XAMPP likely uses Apache → multiple PHP workers → feels faster.

   Fix options:
   - Use XAMPP Apache pointed at backend/public (recommended for local dev)
   - Or Laragon / Laravel Herd
   - Or enable workers in .env: PHP_CLI_SERVER_WORKERS=4
   - Run: php artisan serve --no-reload


2. FRONTEND DUPLICATE FETCHES (StudentsPage.jsx)
   ----------------------------------------------------------------
   Two useEffect hooks both fetch on page load:

   File: frontend/src/features/student-affairs/pages/StudentsPage.jsx

   - Effect on [search] → debounced fetch after 380ms, resets page to 1
   - Effect on [page]   → fetch immediately on mount

   On load:
     1. page effect  → fetch immediately
     2. search effect → fetch again after 380ms
     3. React StrictMode (main.jsx) → both run TWICE in dev

   Result: up to 6 identical GET /students calls (matches HAR).


3. STUDENT PROFILE LOADS EVERYTHING UPFRONT
   ----------------------------------------------------------------
   File: frontend/src/features/student-affairs/pages/StudentProfilePage.jsx

   Promise.all loads profile + transcript + cgpa + attendance + years + semesters
   even if user only views "Personal info" tab.

   transcript, cgpa, attendance are HEAVY backend services.
   Fine alone; painful when all fire at once on single-thread server.


4. DASHBOARD HOME (StudentAffairsHome.jsx) — 4 PARALLEL CALLS
   ----------------------------------------------------------------
   File: frontend/src/features/student-affairs/pages/StudentAffairsHome.jsx

   On load:
     - GET /students?per_page=1        (only for total count)
     - GET /colleges?per_page=100
     - GET /departments?per_page=100
     - GET /academic-programs?per_page=100

   Then frontend joins colleges + departments + programs in JavaScript.
   Could be ONE backend endpoint instead.


5. NOT A DATABASE SIZE PROBLEM
   ----------------------------------------------------------------
   Only 2 students in DB during test.
   Even php artisan tinker took ~2s to boot → PHP/Laravel bootstrap cost,
   not MySQL struggling on row count.
